# Requirements Document

## Introduction

This specification covers three enhancements to the TimeFlow admin pages: adding a completeness filter to the Timesheet Submission page, restricting edit/delete actions on approved users in the Master Data User page, and making the Code and Role fields auto-generated and read-only in the Add/Edit User form. These changes improve admin workflow efficiency and enforce data integrity rules for approved user records.

## Glossary

- **Portal**: The TimeFlow web application front-end built with Laravel/Blade templates
- **Admin_User**: A user with userType `admin` who manages platform configuration, users, and projects
- **Super_Admin**: A user with userType `superadmin` who has full system access including admin management
- **API_Service**: The AWS AppSync GraphQL API that serves as the backend data layer
- **Submission_Page**: The admin page displaying all employee timesheet submissions for a selected period, with columns for week period, user name, total hours, status, and actions
- **User_Management_Page**: The Master Data page where Admin_User or Super_Admin manages user records, displaying user code, name, email, role, status, and action icons
- **User_Form**: The modal dialog for adding or editing a user record, containing fields for Code, Full Name, Email, Role, Position, and Department
- **Complete_Submission**: A timesheet submission where the employee has logged 40 or more total hours for the period
- **Incomplete_Submission**: A timesheet submission where the employee has logged fewer than 40 total hours for the period
- **Approved_User**: A user record with status set to `active` that has been approved by an Admin_User or Super_Admin
- **User_Code**: A system-generated unique identifier for a user record following the pattern `USR-NNN` (e.g., USR-004)

## Requirements

### Requirement 1: Timesheet Submission Completeness Filter

**User Story:** As an Admin_User, I want to filter timesheet submissions by completeness (complete vs incomplete), so that I can quickly identify employees who have not logged sufficient hours for the period.

#### Acceptance Criteria

1. THE Submission_Page SHALL display a filter control allowing the Admin_User to select one of three options: All Submissions, Complete Submissions, or Incomplete Submissions
2. WHEN the Admin_User selects "Complete", THE Submission_Page SHALL display only submissions where totalHours is 40 or greater
3. WHEN the Admin_User selects "Incomplete", THE Submission_Page SHALL display only submissions where totalHours is less than 40
4. WHEN the Admin_User selects "All", THE Submission_Page SHALL display all submissions regardless of totalHours
5. THE Submission_Page SHALL default the completeness filter to "All" when the page loads
6. WHEN the Admin_User changes the completeness filter, THE Submission_Page SHALL update the displayed submissions without a full page reload
7. THE Submission_Page SHALL update the submission count indicator to reflect the number of filtered results

### Requirement 2: Restrict Actions on Approved Users

**User Story:** As an Admin_User, I want the system to prevent editing or deleting approved users, so that approved user records maintain data integrity and are not accidentally modified.

#### Acceptance Criteria

1. WHILE a user record has status of `active` (Approved_User), THE User_Management_Page SHALL hide the edit action icon for that user row
2. WHILE a user record has status of `active` (Approved_User), THE User_Management_Page SHALL hide the delete action icon for that user row
3. WHILE a user record has status other than `active`, THE User_Management_Page SHALL display both the edit and delete action icons for that user row
4. IF an Admin_User attempts to call the updateUser API mutation for an Approved_User, THEN THE API_Service SHALL reject the request and return an error indicating that approved users cannot be edited
5. IF an Admin_User attempts to call the deleteUser API mutation for an Approved_User, THEN THE API_Service SHALL reject the request and return an error indicating that approved users cannot be deleted

### Requirement 3: Auto-Generated Read-Only User Code

**User Story:** As an Admin_User, I want the user Code field to be auto-generated by the system, so that user codes follow a consistent pattern and cannot be manually overridden.

#### Acceptance Criteria

1. WHEN the Admin_User opens the User_Form to add a new user, THE Portal SHALL display the Code field as read-only with a system-generated value following the pattern `USR-NNN`
2. THE API_Service SHALL generate the User_Code by determining the next sequential number based on existing user records
3. WHEN the Admin_User opens the User_Form to edit an existing user, THE Portal SHALL display the Code field as read-only showing the existing User_Code
4. THE User_Form SHALL render the Code field with a visually distinct read-only style (grey background) to indicate the field is not editable
5. IF the Admin_User submits the User_Form, THEN THE Portal SHALL exclude the Code field from the form payload, relying on the API_Service to assign the code

### Requirement 4: Auto-Generated Read-Only Role Field

**User Story:** As an Admin_User, I want the Role field to be auto-generated based on the user type being created, so that role assignment is consistent and cannot be manually overridden.

#### Acceptance Criteria

1. WHEN the Admin_User opens the User_Form to add a new user (Employee), THE Portal SHALL display the Role field as read-only with the value "Employee"
2. WHEN the Admin_User opens the User_Form to edit an existing user, THE Portal SHALL display the Role field as read-only showing the existing role value
3. THE User_Form SHALL render the Role field with a visually distinct read-only style (grey background) to indicate the field is not editable
4. THE API_Service SHALL assign the role value `Employee` for users created by an Admin_User (userType: user)
5. IF the Admin_User submits the User_Form, THEN THE Portal SHALL exclude the Role field from the editable form payload, relying on the API_Service to assign the role based on userType
